# 技能库需求

## 目标

技能库要让用户看懂自己有哪些 AI 技能、每个技能做什么、从哪里来、哪个 AI 工具可以使用、怎么使用、哪些需要处理。

普通用户默认看到简单列表和清楚解释。高级信息只在展开详情或高级区域出现。

## 列表视图

每个技能卡片必须显示：

- 技能名称。
- 一句话用途。
- 来源。
- 可用工具，例如 Codex、Claude 或 Agents。
- 状态。
- 使用入口，例如 `$skill-name`。
- 是否需要处理。

列表顶部必须显示：

- 技能总数。
- 需要处理的数量。
- 搜索框。
- 来源筛选。
- 状态筛选。

## 详情视图

技能详情必须显示：

- 名称。
- 原始名称。
- 它能做什么。
- 适合什么时候用。
- 哪个 AI 工具可以使用它。
- 如何使用。
- 来源。
- 状态。
- 相关问题。
- 高级来源信息。

详情页不能执行技能，不能修改技能文件，不能自动复制到其他工具。

## 分组

默认浏览按任务和功能分类，例如写作、代码、设计、排查、资料整理和其他用途。

来源只作为筛选、详情来源说明和高级来源视图，不能替代默认的功能分类骨架。

高级用户可以在高级区域切换为更细的来源视图，例如 Codex、Claude、Agents、本地共享、项目里的技能、全局来源、手动选择的文件夹、插件带来的技能和来源不明。普通用户默认不需要看到完整路径。

## 搜索

搜索应覆盖：

- 技能名称。
- 原始名称。
- 用途说明。
- 来源名称。
- 可用工具。
- 使用入口。

搜索不执行任何命令。

## 筛选

必须支持：

- 全部。
- 可用。
- 需要处理。
- 重复。
- 已损坏。
- 来源不明。

可选支持：

- 按来源筛选。
- 按工具筛选，例如 Codex、Claude、Agents。

## 状态标签

| 状态 | 含义 |
| --- | --- |
| 可用 | 找到了说明和来源，可以使用。 |
| 需要处理 | 有问题需要用户查看。 |
| 重复 | 发现多个相同或近似的技能。 |
| 已损坏 | 来源缺文件、格式异常或无法读取。 |
| 来源不明 | 找到了线索，但还不能判断来自哪里。 |
| 未检查 | 还没有完成查找或来源未被打开。 |

## 来源标签

来源标签必须用普通用户能理解的词：

- Codex。
- Claude。
- Agents。
- 全局来源。
- 项目来源。
- 本地共享。
- 项目文件夹。
- 插件。
- 手动添加。
- 来源不明。

高级详情可以显示路径提示，但应隐藏敏感部分。

## 使用说明

每个可用技能都必须提供至少一种使用方式：

- 在 Codex 中输入的技能名。
- 在 Claude、Agents 或其他工具中的调用方式。
- 哪个工具可以使用它。
- 可复制的简短提示。
- 什么时候适合用。
- 不适合什么时候用。

如果无法判断使用方式，显示：

- 暂时无法判断使用方法。请在高级信息里查看来源。

## 异常处理

重复技能：

- 默认只展示推荐项。
- 在详情里说明还找到了哪些重复来源。
- 不自动删除或合并。

已损坏技能：

- 显示缺少什么，例如“找不到说明文件”。
- 提供人工检查建议。
- 不自动修复。

来源不明：

- 放到“来源不明”分组。
- 说明 AIOS Desktop 还不能判断它是不是完整技能。
- 不把它计入“可用技能”。

无法读取：

- 显示为“需要处理”。
- 说明可能是权限问题或文件已移动。
- 不要求用户修改系统权限。

## 作用域契约

技能的作用域只能来自显式的产品级元数据，不能从路径、文件夹名称、来源应用、source_kind 或缺失的字段推断。

### 作用域来源

- `global`：仅当内置扫描模板显式声明全局产品作用域，或用户显式配置为全局且已确认时。
- `project`：仅当扫描来源请求显式包含稳定的 `projectId`，并且该项目 ID 已存在于项目注册表中时。
- `unknown`：默认值。所有未提供显式作用域元数据的来源都属于未整理作用域；当前所有历史来源都可以继续保持 `unknown`，直到用户或权威产品配置显式赋值。

### 禁止的推断规则

- 禁止从绝对路径推断项目身份。
- 禁止从文件夹 basename 生成项目显示名称。
- 禁止将来源应用名称（Codex、Claude、Cursor 等）当作作用域。
- 禁止将 `source_kind` 当作作用域。
- 禁止将用户 home 目录视为全局作用域。
- 禁止仅因存在 `project_label` 文本就将其作为项目作用域证据。
- 禁止将 legacy `resourceCorpus.projectMap` 作为普通用户的产品级项目模型。

### 项目身份

- 项目 ID 必须是稳定、持久、由产品级配置或用户显式确认生成的标识符；**项目显示名称不是身份**。
- 项目显示名称必须由用户确认或显式配置，不能从路径派生；重命名显示名称不得改变项目 ID。
- 项目记录通过独立的项目注册表维护，不自动导入历史 `project_label` 值。
- 禁止仅因两个项目的显示名称相同就合并它们；相同标签必须对应不同的稳定项目 ID。

### 显式作用域赋值与更新语义

扫描来源的作用域必须通过显式产品命令赋值，并支持重新分配和清除：

- 省略 `scope_kind`、`project_id`、`scope_source`、`scope_confirmed` 等字段时，必须保留现有作用域赋值。
- 显式赋值为 `global` 时，必须清除已关联的 `project_id`。
- 显式赋值为 `project` 时，必须提供已存在的稳定 `project_id`。
- 显式赋值为 `unknown` 时，必须清除已关联的 `project_id`，使来源回到未整理状态。
- 将来源从项目 A 改为项目 B 时，必须用新项目 ID 替换旧项目 ID。
- 无效的组合（如 `project` 缺少 `project_id`、`global` 携带 `project_id`、`unknown` 携带 `project_id`、不存在的 `project_id`）必须在持久化前被拒绝，且不得改变现有来源状态。
- 作用域赋值不得触发扫描。

### 受控的 scopeSource

`scope_source` 只接受受控值，用于说明作用域决策的权威来源：

- `userConfig`：通过显式产品命令由用户配置。
- `builtinProfile`：来自声明了作用域的受信任内置扫描模板元数据。
- `legacyMigration`：来自历史数据迁移，**不能**作为已确认的作用域证据。
- `unknown`：未整理来源，**不能**作为已确认的作用域证据。
- 任何无法识别的持久化值必须安全降级为 `legacyMigration` 且未确认。
- `scopeConfirmed` 为 `true` 时，`scope_source` 必须是 `userConfig` 或 `builtinProfile`。

### 多源作用域摘要

一个去重后的技能可能同时具有以下证据：

- `hasGlobalSource`：存在至少一个已确认的全局来源。
- `projects`：技能所属的一个或多个稳定项目引用。
- `hasUnknownSource`：存在未整理来源的证据，即使同时存在显式来源也不得丢弃。

分类规则：

- `globalOnly`：至少一个已确认的全局来源，且没有项目来源和未整理来源。
- `projectOnly`：至少一个已确认的项目来源，且没有全局来源和未整理来源。
- `mixed`：同时存在已确认的全局来源和项目来源，或存在多个已确认的项目来源。
- `unknown`：只有未整理来源。

`hasUnknownSource` 单独保留未整理证据的存在；显式来源与未整理证据共存时，分类仍按显式来源决定，但 `hasUnknownSource` 为 `true`。

### 接口契约

- `SkillListItem` 和 `SkillDetail` 必须返回一致的多源作用域摘要。
- TypeScript 契约必须与 Rust 序列化字段完全一致（camelCase）。
- 本阶段（P6D-B）只建立持久化与数据契约，不完成 Skills 界面重设计。
