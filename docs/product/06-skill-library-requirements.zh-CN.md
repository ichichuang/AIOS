# 技能库需求

## 目标

技能库要让用户看懂自己有哪些 AI 技能、每个技能做什么、从哪里来、哪个 AI 工具可以使用、怎么使用、哪些需要处理。

普通用户默认看到简单列表和清楚解释。高级信息只在展开详情或高级区域出现。

## 列表视图

每个技能卡片必须显示：

- 技能名称。
- 一句话用途。
- 来源。
- 可用工具，例如 Codex、Claude 或 Agents。
- 状态。
- 使用入口，例如 `$skill-name`。
- 是否需要处理。

列表顶部必须显示：

- 技能总数。
- 需要处理的数量。
- 搜索框。
- 来源筛选。
- 状态筛选。

## 详情视图

技能详情必须显示：

- 名称。
- 原始名称。
- 它能做什么。
- 适合什么时候用。
- 哪个 AI 工具可以使用它。
- 如何使用。
- 来源。
- 状态。
- 相关问题。
- 高级来源信息。

详情页不能执行技能，不能修改技能文件，不能自动复制到其他工具。

## 分组

默认浏览按任务和功能分类，例如写作、代码、设计、排查、资料整理和其他用途。

来源只作为筛选、详情来源说明和高级来源视图，不能替代默认的功能分类骨架。

高级用户可以在高级区域切换为更细的来源视图，例如 Codex、Claude、Agents、本地共享、项目里的技能、全局来源、手动选择的文件夹、插件带来的技能和来源不明。普通用户默认不需要看到完整路径。

## 搜索

搜索应覆盖：

- 技能名称。
- 原始名称。
- 用途说明。
- 来源名称。
- 可用工具。
- 使用入口。

搜索不执行任何命令。

## 筛选

必须支持：

- 全部。
- 可用。
- 需要处理。
- 重复。
- 已损坏。
- 来源不明。

可选支持：

- 按来源筛选。
- 按工具筛选，例如 Codex、Claude、Agents。

## 状态标签

| 状态 | 含义 |
| --- | --- |
| 可用 | 找到了说明和来源，可以使用。 |
| 需要处理 | 有问题需要用户查看。 |
| 重复 | 发现多个相同或近似的技能。 |
| 已损坏 | 来源缺文件、格式异常或无法读取。 |
| 来源不明 | 找到了线索，但还不能判断来自哪里。 |
| 未检查 | 还没有完成查找或来源未被打开。 |

## 来源标签

来源标签必须用普通用户能理解的词：

- Codex。
- Claude。
- Agents。
- 全局来源。
- 项目来源。
- 本地共享。
- 项目文件夹。
- 插件。
- 手动添加。
- 来源不明。

高级详情可以显示路径提示，但应隐藏敏感部分。

## 使用说明

每个可用技能都必须提供至少一种使用方式：

- 在 Codex 中输入的技能名。
- 在 Claude、Agents 或其他工具中的调用方式。
- 哪个工具可以使用它。
- 可复制的简短提示。
- 什么时候适合用。
- 不适合什么时候用。

如果无法判断使用方式，显示：

- 暂时无法判断使用方法。请在高级信息里查看来源。

## 异常处理

重复技能：

- 默认只展示推荐项。
- 在详情里说明还找到了哪些重复来源。
- 不自动删除或合并。

已损坏技能：

- 显示缺少什么，例如“找不到说明文件”。
- 提供人工检查建议。
- 不自动修复。

来源不明：

- 放到“来源不明”分组。
- 说明 AIOS Desktop 还不能判断它是不是完整技能。
- 不把它计入“可用技能”。

无法读取：

- 显示为“需要处理”。
- 说明可能是权限问题或文件已移动。
- 不要求用户修改系统权限。

## 作用域契约

技能的作用域只能来自显式的产品级元数据，不能从路径、文件夹名称、来源应用、source_kind 或缺失的字段推断。

### 作用域来源

- `global`：仅当内置扫描模板显式声明全局产品作用域，或用户显式配置为全局且已确认时。
- `project`：仅当扫描来源请求显式包含稳定的 `projectId`，并且该项目 ID 已存在于项目注册表中时。
- `unknown`：默认值。所有未提供显式作用域元数据的来源都属于未整理作用域；当前所有历史来源都可以继续保持 `unknown`，直到用户或权威产品配置显式赋值。

### 禁止的推断规则

- 禁止从绝对路径推断项目身份。
- 禁止从文件夹 basename 生成项目显示名称。
- 禁止将来源应用名称（Codex、Claude、Cursor 等）当作作用域。
- 禁止将 `source_kind` 当作作用域。
- 禁止将用户 home 目录视为全局作用域。
- 禁止仅因存在 `project_label` 文本就将其作为项目作用域证据。
- 禁止将 legacy `resourceCorpus.projectMap` 作为普通用户的产品级项目模型。

### 项目身份

- 项目 ID 必须是稳定、持久、由产品级配置或用户显式确认生成的标识符；**项目显示名称不是身份**。
- 项目显示名称必须由用户确认或显式配置，不能从路径派生；重命名显示名称不得改变项目 ID。
- 项目记录通过独立的项目注册表维护，不自动导入历史 `project_label` 值。
- 禁止仅因两个项目的显示名称相同就合并它们；相同标签必须对应不同的稳定项目 ID。

### 显式作用域赋值与更新语义

扫描来源的作用域必须通过显式产品命令赋值，并支持重新分配和清除：

- 省略 `scope_kind`、`project_id`、`scope_source`、`scope_confirmed` 等字段时，必须保留现有作用域赋值。
- 显式赋值为 `global` 时，必须清除已关联的 `project_id`。
- 显式赋值为 `project` 时，必须提供已存在的稳定 `project_id`。
- 显式赋值为 `unknown` 时，必须清除已关联的 `project_id`，使来源回到未整理状态。
- 将来源从项目 A 改为项目 B 时，必须用新项目 ID 替换旧项目 ID。
- 无效的组合（如 `project` 缺少 `project_id`、`global` 携带 `project_id`、`unknown` 携带 `project_id`、不存在的 `project_id`）必须在持久化前被拒绝，且不得改变现有来源状态。
- 作用域赋值不得触发扫描。

### 受控的 scopeSource

`scope_source` 只接受受控值，用于说明作用域决策的权威来源：

- `userConfig`：通过显式产品命令由用户配置。
- `builtinProfile`：来自声明了作用域的受信任内置扫描模板元数据。
- `legacyMigration`：来自历史数据迁移，**不能**作为已确认的作用域证据。
- `unknown`：未整理来源，**不能**作为已确认的作用域证据。
- 任何无法识别的持久化值必须安全降级为 `legacyMigration` 且未确认。
- `scopeConfirmed` 为 `true` 时，`scope_source` 必须是 `userConfig` 或 `builtinProfile`。

### 多源作用域摘要

一个去重后的技能可能同时具有以下证据：

- `hasGlobalSource`：存在至少一个已确认的全局来源。
- `projects`：技能所属的一个或多个稳定项目引用。
- `hasUnknownSource`：存在未整理来源的证据，即使同时存在显式来源也不得丢弃。

分类规则：

- `globalOnly`：至少一个已确认的全局来源，且没有项目来源和未整理来源。
- `projectOnly`：至少一个已确认的项目来源，且没有全局来源和未整理来源。
- `mixed`：同时存在已确认的全局来源和项目来源，或存在多个已确认的项目来源。
- `unknown`：只有未整理来源。

`hasUnknownSource` 单独保留未整理证据的存在；显式来源与未整理证据共存时，分类仍按显式来源决定，但 `hasUnknownSource` 为 `true`。

### 接口契约

- `SkillListItem` 和 `SkillDetail` 必须返回一致的多源作用域摘要。
- TypeScript 契约必须与 Rust 序列化字段完全一致（camelCase）。
- P6D-B 只建立持久化与数据契约，不完成 Skills 界面重设计。

## 能力分类与浏览模型（P6D-C）

### 设计原则

- 能力分类必须基于用户想完成的任务，不能基于来源应用、提供者名称、路径或技术实现细节。
- 来源、提供者、工具兼容性、路径、作用域和状态只能作为次要信息或次要筛选，不能取代能力作为首要分组。
- 分类必须有确定性的顺序、可预测的未知状态，并且不对弱信号假装确定。

### 正式能力分类

技能库使用 9 个正式能力分类和 1 个明确的未知分类，顺序固定如下：

1. **前端与界面**（`frontend-ui`）：Web、小程序/移动端 UI、前端框架、组件、布局与样式实现。
2. **通用代码开发**（`coding`）：通用编程、算法、语言级开发与日常实现辅助。
3. **代码审查与重构**（`code-review`）：代码审查、重构、lint、类型检查、AST 操作与 codemod。
4. **测试与质量**（`testing-qa`）：单元测试、集成测试、浏览器测试、E2E、截图、回归验证与质量检查。
5. **文档与写作**（`docs-writing`）：文档、Markdown、知识库、RAG 内容、写作、编辑与润色。
6. **数据与自动化**（`data-automation`）：数据处理、表格、CSV、SQL、JSON、自动化、脚本、构建、流水线与 CI/CD。
7. **研究与分析**（`research-analysis`）：研究、调研、综合、总结、比较、方法论与分析工作。
8. **安全与治理**（`security-governance`）：安全审查、隐私、风险、策略、治理、依赖安全与护栏。
9. **设计与视觉**（`design-visual`）：视觉设计、主题、设计系统、Figma、图像生成、动画、动效与视觉打磨。
10. **尚未分类**（`unknown`）：没有足够可靠的能力证据，暂时无法归入稳定分类。

已移除的旧分类包括：`automation-scripts`、`data-spreadsheets`、`browser-debug`、`mcp-integrations`、`project-governance`、`local-system`、`other`。

### 分类证据规则

- 允许使用的证据：`displayName`、`originalName`、`shortPurpose`、`usageText`、`tags`、`capabilities` 以及 Skill 条目上其他明确的商品描述字段。
- 禁止使用的证据：`sourceLabel`、`sourceKindLabel`、`primaryPathHint`、项目标签、`scopeSummary`、`availableInTools`、提供者名称（Codex、Claude、Cursor、Agents、MCP 等） merely because they are providers。
- 提供者名称只有在构成实际能力短语的一部分、且同时存在其他非提供者能力信号时，才能作为辅助证据；单独的提供者名称必须导致 `unknown`。
- 弱通用词（如 tool、helper、project、AI、local、path、service）不能单独决定分类。
- 每个 Skill 必须有且只有一个主分类；最多保留两个次级分类标签。
- 分类置信度为 `high`、`medium`、`low` 或 `unknown`；低于最小证据阈值的 Skill 必须归入 `unknown`。
- 平局时按固定分类顺序打破，禁止依赖对象迭代顺序。

### 作用域浏览成员语义

技能页提供两个独立维度：作用域与能力。作用域浏览使用 `scopeSummary` 中的显式证据，禁止重新从路径、来源或标签推断。

- **全部技能**（`all`）：包含所有 Skill，每个 Skill 只出现一次。
- **全局技能**（`global`）：包含 `hasGlobalSource === true` 的 Skill，包括 `globalOnly` 和 `mixed`。
- **项目技能**（`project`）：包含 `projects.length > 0` 的 Skill，包括 `projectOnly` 和 `mixed`。
- **范围未整理**（`unknown`）：只包含 `classification === "unknown"` 的 Skill；`mixed` Skill 即使 `hasUnknownSource === true` 也不得放入此作用域。

`mixed` Skill 同时属于全局和项目两个视图；其 `hasUnknownSource` 保留为次要事实，但不改变作用域归属。

### 项目选择器

- 项目选项从所有 Skill 的 `scopeSummary.projects` 聚合而来。
- 去重必须基于稳定的 `projectId`，禁止仅按 `projectLabel` 合并。
- 选项按 `projectLabel` 排序，`projectLabel` 相同时按 `projectId` 排序；相同标签但不同 ID 的项目必须作为独立选项显示。
- 使用 `projectId` 作为选择值，使用 `projectLabel` 作为显示文本；禁止将路径或 ID 渲染为项目身份。
- 项目选择器仅在“项目技能”作用域下显示；没有已登记项目时显示禁用状态并提示“暂无已登记项目”。

### 能力与作用域的组合

- 默认能力选择为“全部能力”。
- 选择“全部能力”时，结果按固定分类顺序分组显示。
- 选择单个能力时，只显示该能力的 Skill。
- “尚未分类”始终排在最后。
- 分类计数在作用域、项目、搜索、状态、来源筛选之后、能力筛选之前计算。
- 搜索覆盖名称、原始名称、用途、使用说明、标签、能力、可用工具和 `sourceLabel`（仅作为可搜索的次要来源），但搜索不改变能力分类。
- 排序规则：先按状态等级（可用优先），再按显示名称的本地化顺序，最后按稳定 ID。

### 次要筛选

- 状态筛选和来源筛选必须置于“更多筛选”区域，不能和能力/作用域选择器争夺视觉注意力。
- 禁止将来源或提供者作为首要分组；禁止创建来源卡片 rail 或提供者分区。

### 选择清除与持久化

- 切换作用域、项目或能力时，必须清除当前选中的 Skill 详情。
- 搜索和次要筛选只有在当前选中的 Skill 仍然可见时才可保留选择，否则清除。
- 使用 `sessionStorage` 持久化 `aios.skills.scope`、`aios.skills.project` 和 `aios.skills.capability`；非法或缺失值必须安全回退，禁止持久化路径或来源派生的项目身份。

### 空状态

- 没有 Skill：`还没有找到 AI 技能` / `开始查找后，这里会显示技能名称、用途和能力分类。`
- 全局作用域为空：`还没有明确归为全局的技能` / `只有经过显式范围确认的技能才会显示在这里。`
- 项目作用域无已登记项目：`还没有已登记的项目技能` / `只有明确归入项目的技能才会显示在这里。`
- 选中项目为空：`这个项目下还没有技能` / `选择其他项目，或查看全部技能。`
- 范围未整理为空：`没有范围未整理的技能` / `当前技能都已有明确的全局或项目范围。`
- 能力分类为空：`这个能力分类下没有技能` / `选择其他能力分类，或清除搜索和筛选。`
- 搜索无结果：`没有匹配结果` / `换一个关键词，或清除筛选后再试。`

### 阶段边界

- P6D-C 建立能力分类、浏览模型与选择器行为，不完成行、检查器、响应式与视觉重设计。
- P6D-D 负责 Skill 行、详情检查器、响应式布局与最终视觉验收。

## P6D-D 最终 Skills 页面、行与详情契约

### 页面层级（从上到下、从重要到次要）

1. **页面标题区**
   - 标题：技能。
   - 描述：按任务和能力浏览已整理的 AI 技能，快速找到适合当前工作的技能。
   - 禁止大型 Hero、摘要卡片、工程术语和悬停动画。

2. **作用域导航**
   - Tab：全部技能、全局技能、项目技能、范围未整理。
   - 每个 Tab 显示数量；当前作用域视觉突出。
   - 保持键盘可访问（role="tablist" / role="tab"）。
   - 作用域的视觉权重必须高于来源和状态筛选。

3. **项目选择器**
   - 仅在“项目技能”作用域下显示。
   - 首项：全部项目。
   - 其他选项按稳定 projectId 去重，显示 projectLabel。
   - 相同 projectLabel 但不同 projectId 必须作为独立选项。
   - 禁止显示路径或从路径推断项目身份。
   - 无已登记项目时显示禁用选择器，提示“暂无已登记项目”。

4. **能力导航**
   - 宽屏使用左侧 rail，首项“全部能力”，随后是 9 个正式分类（固定顺序），最后是“尚未分类”。
   - 每项包含小图标、标签和计数；当前项使用背景加深+左侧强调线，非仅颜色。
   - 零计数保持弱化但仍可读。
   - rail 中不得出现来源、状态、应用、项目或提供者条目。
   - 窄屏将 rail 替换为单个紧凑的 MUI Select / Menu，禁止使用横向滚动 chip。

5. **结果区**
   - 主工作面，独立滚动。
   - 标题、作用域、筛选、能力导航保持固定。
   - “全部能力”时按能力分组显示，每组显示分类名称、简短说明和计数。
   - 组之间用间距、字体和轻分隔线区分，不使用厚重卡片。
   - 选择单个能力时只显示该能力标题和列表，不在每行重复能力 chip。

6. **次要筛选**
   - 状态和来源仅通过“更多筛选”展开。
   - 展开区域显示当前激活筛选计数和“重置筛选”。
   - 来源筛选不得置于作用域或能力之前；禁止按来源分组、创建来源卡片或提供者分区。

### Skill 行信息层级

从上到下：

1. 技能显示名称。
2. 范围标签（根据 scopeSummary 计算）。
3. 仅在 broken/needsAttention 时显示的状态标签（不可用 / 需要检查）。
4. 一句话用途。
5. 适合：使用场景。
6. 兼容 AI 工具（最多两个可见标签，剩余 +N）。
7. 已知来源（次要、柔和）。
8. 可选的次级能力提示（仅在需要时）。

### 范围标签措辞

- globalOnly：全局。
- projectOnly（一个项目）：项目 · {projectLabel}。
- projectOnly / mixed（多个项目且无全局）：{N} 个项目。
- mixed（含全局和项目）：全局 + {N} 个项目。
- unknown：范围未整理。
- hasUnknownSource 与已确认成员共存时，行内只显示已确认范围；详情中说明另有来源尚未整理范围。

### Skill 详情检查器层级

1. 标题区
   - 能力分类（如“前端与界面”）。
   - 技能显示名称。
   - 范围摘要标签。
   - 仅在 broken/needsAttention 时显示的状态标签。
   - 关闭控制（由 shell 提供）。

2. 主体区（按顺序）
   - 它能做什么。
   - 适合什么时候用。
   - 如何使用。
   - 可在哪些 AI 工具中使用（仅显式 availableInTools，不含 Unknown；为空时隐藏）。
   - 归属范围（按 classification 列出全局 / 项目 / 未知；hasUnknownSource 时附加说明）。

3. 缺失信息
   - 多个主要字段同时缺失时，只显示一次：这个技能的详细说明还不完整，AIOS 只展示当前已记录的信息。

4. 折叠区（默认折叠）
   - 来源与记录：来源标签、来源类型、可用工具、记录时间、重复来源；不展示私有绝对路径。
   - 检查信息：仅在 broken / attention 等需要时展开。
   - 高级信息：原始技术名、别名、标签、能力、来源数、安全边界；默认折叠。

### 来源与状态的次级地位

- 普通 available 状态不在行内重复显示“可用”。
- 不在行内重复显示“来源不明” chip。
- 不在行内重复显示原始技术名、路径、ID、重复计数、token 估计或扫描器信息。
- 提供者名称、来源应用、工具类型不得作为能力分类标题或默认分组。

### 响应式与滚动所有权

- 1280px 及以上：能力 rail 约 200–224px，结果区灵活，检查器约 340–380px；结果区和检查器各自独立滚动。
- 1024px：rail 收窄，行内元数据安全换行，检查器可用。
- 900px 检查器打开：沿用 P5E-C 保护，主内容不得塌成 rail 宽列；检查器关闭后保留选中行上下文。
- 最窄桌面宽度：能力 rail 退化为紧凑 Select，作用域 tab 和项目选择器可用，元资料自然堆叠，无水平滚动、无重复滚动条。
- Shell 固定不滚动；结果面板与检查器各自滚动；能力导航保持固定。

### 动效限制

- 仅允许 background、border、opacity 过渡。
- 禁止对静态容器、行、卡片使用 transform、translate、scale、lift、rotate、resize 或阴影扩散悬停。
- 禁止引入新的动画库、GSAP、玻璃效果、嵌套卡片表面、装饰阴影、提供者 logo、大插画、徽章墙。
