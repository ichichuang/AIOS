# 信息架构

## 新主导航

AIOS Desktop 的主导航只保留四项：

| 导航 | 面向谁 | 主要问题 |
| --- | --- | --- |
| 首页 | 所有人 | 我有多少 AI 技能和 MCP 工具？哪些需要处理？ |
| 技能 | 所有人 | 每个技能做什么？从哪里来？哪个工具能用？怎么用？ |
| MCP | 所有人 | 哪些 MCP 服务和工具可用？在哪里配置？哪里坏了？怎么处理？ |
| 高级 | 开发者和需要排查的人 | 来源、问题细节、隐私控制和开发者信息在哪里？ |

## 首页

首页是普通用户的默认入口。

必须包含：

- 技能总数。
- MCP 服务和工具总数。
- 需要处理的数量。
- 最近一次查找时间。
- 一键查找按钮。
- 手动选择文件夹入口。
- 简短隐私说明。

首页不能展示大段内部日志或历史模块。

## 技能

技能页是主要产品页面。

必须包含：

- 技能列表。
- 搜索。
- 来源筛选。
- 状态筛选。
- 可用工具说明。
- 技能详情。
- 使用说明。

## MCP

MCP 页是第二个主要产品页面。

必须包含：

- MCP 服务列表。
- 每个服务下的工具列表。
- 来源。
- 配置位置说明。
- 健康状态。
- 需要处理的问题。
- 用户可理解的修复建议。

## 高级

高级页只服务排查和来源管理。

可以包含：

- 查找位置管理。
- 手动添加文件夹。
- 来源分组：全局来源、项目来源、Codex、Claude、手动选择的文件夹。
- 跳过项摘要。
- 错误摘要。
- 隐私和本地数据控制。
- 开发者补充信息。

高级页不能变成默认首页，也不能把内部工程概念推给普通用户。

## 必须移出主导航的内容

以下内容不是一级产品模块，不能出现在主导航：

- Scripts
- Reports
- Policies
- Validators
- Legacy
- raw scan diagnostics
- internal scanner/debug concepts

处理方式：

- Scripts：只有当它能解释某个技能或 MCP 的使用方式时，放在详情页或高级说明中。
- Reports：只作为高级里的“历史说明”或“问题摘要来源”，不做主模块。
- Policies：不要作为用户模块展示；如需表达，改成“安全说明”或“隐私说明”。
- Validators：不要作为用户模块展示；如需表达，改成“检查结果”。
- Legacy：不要作为用户入口；历史示例只可放在高级的开发者说明里。
- raw scan diagnostics：不要展示原始诊断；改成“查找问题摘要”。
- internal scanner/debug concepts：只放在开发者附录或高级排查中。

## 旧查找管理入口的处理

旧的查找管理入口不再是主导航。普通用户通过首页的“开始查找”和“选择文件夹”完成流程。

高级用户可以在“高级”里进入“查找位置管理”。

## 旧文档和旧界面的状态

当前 README、设计系统和前端配置中仍记录了更宽的模块结构。新的产品规划以本目录为准。后续实现时应以四项主导航为目标，不在本阶段修改 UI。
